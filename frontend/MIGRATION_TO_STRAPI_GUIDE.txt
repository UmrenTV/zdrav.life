Below is the setup I would use for zdrav.life.

1. What you should build

Use this architecture:

Frontend: your existing Next.js site remains the presentation layer.

Backend CMS: separate Strapi v5 project for content, media, SEO fields, products, reviews, comments, home page sections, legal pages, and site settings.

Bridge layer in Next.js: keep your existing lib/data/services abstraction, but start swapping JSON readers for Strapi fetchers one domain at a time.

Cursor + MCP: Cursor gets connected to Strapi through MCP so it can inspect schemas, understand content types, and help generate/update backend pieces from inside the editor. Cursor supports MCP, and Cursor’s docs/changelog mention one-click MCP install, OAuth support, and MCP-based tool access.

That matches your PLAN almost perfectly because the PLAN already asked for a future-ready separation between UI and content/data access, plus content types for posts, products, reviews, comments, categories, tags, videos, gallery items, FAQs, testimonials, and site settings.

2. Recommended migration strategy

Do not migrate everything at once.

Best order:

Site settings + homepage content

Blog posts, categories, tags

Authors + SEO components

Videos / Instagram-style media / gallery

Products + product categories + reviews

Legal pages

Comments

Forms / contact submissions

Optional commerce/order layer later

Why this order:

it gives you fast visible wins,

keeps the frontend stable,

and lets you test the Strapi integration on low-risk content first.

Given your current app already has homepage, blog, shop, reviews, comments, media routes, filters, category/tag pages, and SEO plumbing, this migration is mostly a source-of-truth replacement, not a redesign.

7. How to connect the current Next.js app to Strapi

Your current app already has a service/data layer and was built with API swap in mind. The PLAN calls for abstracted data access so local JSON can later be replaced by remote APIs, and the commit message confirms that service methods and API-backed routing were already aligned in that direction.

So do this:

7.1 Keep the interface, swap the source

Example idea:

// current
getPosts(): Promise<Post[]>

// next step
getPosts({ source: "strapi" }): Promise<Post[]>

Then once stable:

getPosts(): Promise<Post[]>

and internally it fetches Strapi.

7.2 Add a mapper layer

Do not let your UI consume raw Strapi responses everywhere.

Create adapters like:

mapStrapiPostToPost()
mapStrapiProductToProduct()
mapStrapiVideoToVideo()
mapStrapiGalleryItemToGalleryItem()
mapStrapiCommentToComment()
mapStrapiReviewToReview()

That protects the frontend from Strapi schema churn.

7.3 Populate relations carefully

Use Strapi queries with populate only where needed. Keep list pages light and detail pages richer.

10. Cursor MCP setup for Strapi

Cursor supports MCP and has docs/changelog indicating MCP integration, one-click install/OAuth support, and HTTP/stdio transports in Cursor contexts.

For Strapi, the cleanest current option I found is the Strapi v5 MCP plugin on Strapi Market: @sensinum/strapi-plugin-mcp. The listing says it supports Strapi v5, exposes a streamable HTTP MCP endpoint, and includes a Cursor .cursor/mcp.json example.

10.1 Install in Strapi

Inside zdravlife-cms:

npm install @sensinum/strapi-plugin-mcp

The plugin page says Strapi v5 auto-discovers it after install, with no extra registration step required.

10.2 Configure it

Create config/plugins.ts in Strapi if you don’t already have it:

export default {
mcp: {
enabled: true,
config: {
session: {
type: "memory",
max: 20,
ttlMs: 600000,
updateAgeOnGet: true,
},
allowedIPs: ["127.0.0.1", "::1"],
},
},
};

The plugin docs show memory sessions and optional IP allowlisting in config/plugins.js or config/plugins.ts.

10.3 Start Strapi
npm run develop
10.4 Add it to Cursor

In your frontend repo or workspace root, create:

{
"mcpServers": {
"strapi": {
"type": "streamable-http",
"url": "http://localhost:1337/api/mcp/streamable"
}
}
}

That exact Cursor configuration pattern is shown on the plugin page, and the endpoint path /api/mcp/streamable is documented there as well.

11. Image generation MCP: what I recommend

I did not find an official OpenAI-hosted “image generation MCP server” product page comparable to Strapi’s plugin. What is official is OpenAI’s image generation API itself: OpenAI’s docs say image generation and editing are available through the Image API and support GPT Image models such as gpt-image-1.5, gpt-image-1, and gpt-image-1-mini, alongside DALL·E models.

So the safest recommendation is:

Recommended

Use a small custom MCP server that wraps the official OpenAI Images API.

That gives you:

full control,

no mystery third-party dependency,

prompts tailored to your exact asset workflow,

and the ability to save generated files directly into your project or upload them into Strapi.

Why not rely blindly on a random community MCP

Because for a serious brand site, the model/API can be official while the MCP wrapper can still be lightweight and yours.

For production I’d use:

Media: object storage

Comments/reviews writes: Next.js server actions or API routes -> Strapi secured endpoints

18. Biggest mistakes to avoid

Do not:

map frontend components 1:1 into Strapi fields with no editorial logic

let raw Strapi response shapes leak across the app

make comments/reviews open public write endpoints

over-model ecommerce before you have real checkout

move every transient UI thing into CMS

build custom Strapi types with inconsistent slugs/naming

store “homepage layout logic” entirely as free-form rich text

Do:

keep CMS for content/data

keep layout logic in code

use reusable components for SEO and CTA fields

use strict slug discipline

keep a migration script folder

document field mappings

19. My recommendation in one sentence

Set up Strapi as a separate v5 backend, install the Strapi MCP plugin for Cursor, keep your Next.js service layer intact, migrate JSON domains one by one into Strapi, and use a small custom image MCP server backed by OpenAI’s official Images API for generating the site’s real media assets.
